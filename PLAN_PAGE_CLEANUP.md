# Plan 页面信息架构清理清单

本清单只用于审计和后续拆分计划。本轮不修改 `PlanView.tsx`、`planPresenter.ts`、engine、数据模型、训练算法或 UI 样式。

## 当前区块列表

| 当前区块 | 位置 | 用户价值 | 当前问题 |
| --- | --- | --- | --- |
| 当前模板 | 主列顶部 | 让用户确认现在训练实际使用哪个模板，尤其是原模板或实验模板。 | 与“实验模板”“版本历史与回滚”重复表达实验模板状态和回滚入口。 |
| 周期时间线 | 主列 | 说明当前周期进度和未来推进节奏。 | 与“计划进度”语义接近，信息价值明确，但不应和建议/草案混排。 |
| 本周训练日 | 主列 | 回答本周每次大致练什么。 | 与“训练日模板”和“当前查看的训练日”都在展示训练日结构，用户需要来回比对。 |
| 训练日模板 | 主列 | 让用户切换查看不同训练日模板。 | 与“本周训练日”重复列出模板名，与“当前查看的训练日”共同组成一个详情流程。 |
| 当前查看的训练日 | 主列 | 展示选中训练日的处方摘要和动作结构。 | 应该是“本周安排”中的详情，而不是独立大区块。 |
| 训练基线 | 侧栏 | 解释训练等级和历史记录对建议保守程度的影响。 | 是辅助上下文，不应抢占计划主流程。 |
| 计划构成 | 侧栏 | 展示主训练、纠偏、功能补丁的计划比例。 | 与本周安排相关，可作为当前计划的折叠辅助信息。 |
| 调整建议 | 侧栏 | 放置尚未生成草案的计划级建议。 | 内部又包含教练建议和训练量建议，导致同类建议重复出现。 |
| 计划相关教练建议 | 调整建议内 | 展示 CoachAction 中与计划相关的待处理建议。 | 与训练量建议、生成草案按钮、草案区状态重复。 |
| 训练量建议 | 调整建议内 | 解释哪些肌群建议增量、维持或减量。 | 与 CoachAction 的“查看训练量建议 / 生成调整草案”重复，容易出现多个相似卡片。 |
| 调整草案 | 侧栏 | 展示已经生成、等待确认应用的草案。 | 是正确的执行区，但需要和“调整建议”严格分层，避免同一建议同时出现在两处。 |
| 实验模板 | 主列底部 | 列出已生成的实验模板，强调不覆盖原模板。 | 与“当前模板”中实验状态、版本历史中的应用记录重复。 |
| 版本历史与回滚 | 主列底部 | 保留应用和回滚记录，提供回滚入口。 | 与“当前模板”的回滚入口和“实验模板”列表重复，适合压缩为历史/复盘信息。 |

## 重复点

1. `本周训练日`、`训练日模板`、`当前查看的训练日` 都在回答“有哪些训练日、每个训练日怎么练”。这三个区块应合并成一个“本周安排”，用列表加详情替代多个平级区块。
2. `调整建议`、`计划相关教练建议`、`训练量建议` 都在展示计划级建议，且可能同时出现“查看训练量建议”“生成调整草案”“待确认”等相似卡片。
3. `生成调整草案` action 和 `调整草案`卡片存在状态重复。草案生成后，原 action 应转为“查看草案”或从待处理建议中移除。
4. `当前模板`、`实验模板`、`版本历史与回滚` 都在表达实验模板和回滚状态。当前状态应归入“当前计划”，历史列表应降级为折叠复盘信息。
5. `训练基线` 和 `计划构成` 是计划解释信息，不是独立任务流。它们可并入“当前计划”的辅助信息或折叠详情。

## 建议合并

| 目标区块 | 合并来源 | 处理方式 |
| --- | --- | --- |
| 当前计划 | 当前模板、训练基线、计划构成、当前实验模板状态、可见回滚入口 | 聚合为 Plan 页第一屏的状态区，只回答“我现在用什么计划”。 |
| 本周安排 | 本周训练日、训练日模板、当前查看的训练日、周期时间线 | 展示训练日顺序和选中详情；周期时间线作为本周安排的上下文。 |
| 待处理建议 | 调整建议、计划相关教练建议、训练量建议中尚未生成草案的项 | 同一来源只保留一个卡片和一个主 CTA；查看类按钮用次级样式。 |
| 调整草案 | 已生成的 ProgramAdjustmentDraft | 只展示真实草案、差异、风险、应用/暂不采用/删除，不混入普通建议。 |

## 最终目标结构

1. 当前计划
   - 当前使用模板。
   - 原模板或实验模板状态。
   - 训练基线和计划构成作为简短辅助信息。
   - 如果正在使用实验模板，显示来源、应用时间、主要调整和回滚入口。

2. 本周安排
   - 本周训练日顺序。
   - 选中训练日的动作和处方摘要。
   - 周期进度只作为安排上下文，不单独制造任务流。

3. 待处理建议
   - 仅显示还没有转成草案的建议。
   - 同一建议只出现一次。
   - 查看类 action 使用“查看训练量建议 / 查看动作进展”等次级按钮。
   - 生成类 action 明确进入“调整草案”。

4. 调整草案
   - 显示草案名称、来源、主要变化、风险、状态和创建时间。
   - 提供“查看差异 / 应用为实验模板 / 暂不采用 / 删除草案”。
   - 已应用草案显示“查看实验模板 / 回滚到原模板”。
   - 已回滚草案只显示状态和复盘信息。

## 迭代顺序

1. 先整理 presenter/view model 命名，把 Plan 页数据分为 `currentPlan`、`weeklySchedule`、`pendingRecommendations`、`adjustmentDrafts` 四组，不改变行为。
2. 合并训练日相关区块：把 `本周训练日`、`训练日模板`、`当前查看的训练日` 收敛为“本周安排”。
3. 合并建议相关区块：去重 `CoachAction`、训练量建议和生成草案入口，确保同一建议只出现一次。
4. 固化草案区职责：草案区只显示真实 `ProgramAdjustmentDraft`，应用实验模板和回滚只在这里或“当前计划”出现。
5. 压缩实验模板和历史：保留回滚安全说明，把历史列表改为辅助复盘信息，不再与当前计划抢主视觉。
6. 更新回归测试：覆盖 Plan 页区块标题、同一建议只显示一次、草案生成后 action 状态转换、实验模板回滚入口仍可用。

## 不允许新增功能的边界

- 不新增饮食模块。
- 不新增后端、登录或云同步。
- 不新增训练算法或计划生成算法。
- 不修改 e1RM、effectiveSet、readiness、progression、warmupPolicy。
- 不修改数据模型。
- 不自动应用草案。
- 不覆盖原模板。
- 不删除实验模板历史或已完成训练历史。
- 不把 Record、My、Training 的职责搬进 Plan 页。
- 不新增页面，不引入新依赖。
