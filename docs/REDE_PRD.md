## 1. 产品定义

### 1.1 一句话

**会带练的力量训练 App**——打开就知道今天练什么、上多少重量、为什么这么安排；它随你的训练表现自动调整，并把长期计划盯住。

品牌主张（文案基线 §1.1）：**先判断，再带练。/ Plan the lift. Show the reason.**

### 1.2 解决什么问题

认真练力量的人最大的问题不是不努力，而是**决策负担**：

- 今天该练还是该休？状态不好要不要硬顶？
- 这个动作该加重量了吗？卡住了怎么办？
- 自己排的计划乱、容易停滞，又不想雇教练。

Rede 把「教练的脑子」装进 App：**决策在前，记录在后**——先告诉你练什么，再帮你记，并随训练表现、恢复线索和长期趋势自动调，每次调整都给理由。

### 1.3 差异化定位

| 竞品类型 | 代表 | 它们的问题 | Rede 的不同 |
|---|---|---|---|
| 手动 logger | Strong、Hevy | 只是本子：练完自己填，决策仍在用户头上 | 决策在前：打开先给今天的判断和处方 |
| 黑箱 AI 生成 | Fitbod | 给计划但不解释为什么，信任难建立 | 每个建议带 Decision Receipt（判断/信号/调整/控制权），可解释、可撤销 |
| 程序库 | Boostcamp、JEFIT | 固定模板，不随个人状态调 | 周期化 + 随表现/恢复自动调整 + 停滞自动退阶 |

定位一句话：**会解释自己的循证教练**（explainable coaching）。

### 1.4 第一版非目标（明确不做）

公开 feed、好友/私信、排行榜、自动发布、账号体系、云同步、watchOS、远程推送、医疗诊断/治疗承诺、按分享解锁功能、把安全必要动作藏进任何付费墙。（系统逻辑 §9.10、§6.4、Master §9、云决策档案）

## 2. 目标用户与场景

### 2.1 主用户画像

认真练力量/增肌（barbell、dumbbell、机械均可）、想持续进步、不想自己当教练排计划的 lifter。目标市场以美国/英语区为主（出海策略见 Roadmap），产品中英双语。训练背景从新手到高阶均可：系统用「校准中 → 初级 → 进阶初期 → 中级 → 高级 → 精英」的整体级别承接（系统逻辑 §6.5.4），自报背景只作冷启动先验。

### 2.2 核心使用场景（一天 / 一周的闭环）

| 时刻 | 用户问题 | Rede 给什么 | 对应页面 |
|---|---|---|---|
| 练前 30 秒 | 今天练不练？练什么？ | 今日裁决 + 处方 + 理由 | 今日 |
| 健身房中 | 这组多少？休多久？下一组呢？ | 逐组带练 + 休息计时 + 下一组建议 | 训练 |
| 练后 | 这次有没有用？ | 记录落盘 + 趋势反馈 | 进展 |
| 每周 | 下周怎么练？计划要不要调？ | 周期结构 + 可回滚调整建议 | 计划 |

## 3. 产品原则（每条需求都要过这八条）

1. **判断先于展示**：每个高频页面先给结论，再给数据；不做 dashboard。
2. **决策在前，记录在后**：系统先提议，用户执行并记录；不是空白表格等用户填。
3. **每个建议带理由**：理由用「信号 + 影响 + 决策」表达（Decision Receipt：Call / Signal / Change / Control），默认收起、可展开。
4. **用户保留控制权**：采纳 / 暂不处理 / 跳过 / 替换 / 撤销随时可见；系统不替用户做最终动作。
5. **本地优先**：数据保存在本机、离线全功能可用、可导出带走；账号和云永远是后置可选项。
6. **诚实数据**：写入不假成功；坏数据先提醒；可信度通过行为表达（判断更保守、显示「正在校准」），**不做置信度 UI 读数**（文案基线 §3.4）。
7. **fitness 不是 medical**：疼痛/不适只触发保守训练调整与「咨询专业人士」建议，绝不诊断、治疗、保证防伤。
8. **双语原生**：中英按各自语言习惯原生撰写，同一意图、两种母语表达，禁逐句机翻。

## 4. 信息架构

底部四 tab + 低频入口（Master §3、AGENTS）：

| 入口 | 使命 | 一句话职责 |
|---|---|---|
| **今日** | 该不该练、练什么、为什么、从哪开始 | 训练前 30 秒的判断 |
| **训练** | 现在这组怎么做、怎么记 | 唯一训练中页面（专注训练） |
| **进展** | 练得有没有效、数据可不可信 | 训练成果的证据页 |
| **计划** | 未来几周怎么练、调整了什么、能不能撤 | 每周教练工作台 |
| Profile / Settings（右上角，非 tab） | 低频设置 | 资料、单位、语言、筛查、HealthKit、导出、订阅 |

## 7. 成功指标

**北极星（Roadmap §5）**：每周完成 ≥2 次有记录训练且看到 ≥1 条教练行动的活跃用户数。

**MVP（TestFlight）阶段——手动观测版**（无远程埋点，靠 TestFlight 数据/反馈/访谈）：

| 指标 | 问题 | 健康信号（初始假设） |
|---|---|---|
| 安装→首训练完成率 | 装了会不会用 | ≥ 40% |
| W1 复练率（首训后 7 天内再练） | 核心价值是否成立 | ≥ 50% |
| 每用户周均记录训练次数 | 是否进入节奏 | ≥ 2 |
| 反馈中「判断可信」正面率 | 差异化是否被感知 | 定性收集 |

> 上表健康信号均为 TestFlight 阶段**手动观测的初始假设**，未经市场验证；进入远程埋点（架构 gate 后）按真实数据校准，不得当作已验证基准引用。

公开上架后启用 Roadmap §5 完整漏斗（激活→留存→试用→转化→LTV）；远程埋点过架构 gate 后才接入。
