# Rede 联动饮食 App — v2 产品设计

> 版本 v2.2（产品团队审核修订版）· 2026-06-03 · 状态：P0 已修订；§6.2 软约束化重审
> 范围：与 Rede 联动的 iOS 原生「饮食记录 + 食材推荐」App
> 营养标准：全部美标（USDA / IOM / FDA）
>
> **v2.2 变更**：§6.2 膳食合理性约束由绝对硬上限改为**软惩罚 + 单条宽松护栏**——避免与硬性宏量约束叠加致无解/脆弱、随体型/目标/餐量自动缩放、保留用户自主。
>
> **v2.1 变更**：修复 6 项 P0（宏量/热量死锁、运动叠加双算、未成年/孕期硬门控、ED 防护阈值、求解器可行性、安全下限挂钩 BMR）+ 关键 P1（单位制、库存档位映射、OCR 五步、数据模型/外部依赖/隐私/验收算例补全）。

## 0. 定位与边界

**一句话**：基于身体数据与目标算出每日营养需求，结合实时运动消耗动态调整，从用户家中现有食材中推荐满足营养目标的「食材 + 克数」清单。

**做什么**：营养目标计算、家庭食材库存管理、按营养目标的食材配比推荐、与 Rede / Apple Watch 联动。

**不做什么**：不推荐菜品/做法（只给食材+克数）；不做医疗/临床营养干预；不替代医生/注册营养师。

**核心原则**：能用权威数据就不靠猜；宁可承认"数据缺失/无解"也不编造；**安全优先于目标达成**。

**准入门控（硬性，见 §8）**：未成年、孕期/哺乳、异常低体重等人群在 onboarding 即被拦截，不进入标准减脂/赤字流程。

## 1. 端到端用户流程

Onboarding（含**单位制选择** + 准入门控）→ 身体档案（身高/体重/性别/年龄）→ 目标（减脂/增肌/保持，可从 Rede 同步）→ 库存初始化（主食标记常备 + 生鲜录入）→ **每日循环**：晨间生成计划 → 进食/记录（采纳即扣库存）→ 运动数据日内校正 → 晚间复盘 → 体重趋势驱动的周度校准。

**单位制（P1，美标必需）**：onboarding 选择 metric / imperial。内部统一以 **kg / cm / g** 存储计算，仅展示层换算 lb / ft-in / oz。所有公式输入先归一为公制。

## 2. 营养目标计算引擎

### 2.1 BMR（Mifflin-St Jeor，仅适用 ≥18 岁）
- 男：`10×kg + 6.25×cm − 5×岁 + 5`
- 女：`10×kg + 6.25×cm − 5×岁 − 161`
- <18 岁：此式不适用 → 走未成年门控（§8）。

### 2.2 计划基线、目标调整与安全下限
- 计划基线 TDEE = `BMR × 1.2`（久坐默认；仅用于晨间排布与无运动数据时的回退，见 §3.1）。
- 目标调整：

| 目标 | 热量调整 | 硬约束 |
|---|---|---|
| 减脂 | −20%（约 −300~−500 kcal） | 赤字 ≤25% **且** 摄入 ≥ 安全下限 |
| 保持 | 0 | — |
| 增肌 | +10~15% | 盈余 ≤20% |

- **安全摄入下限（P0，两条取严）**：`下限 = max(性别绝对值[女1200/男1500], BMR×0.9)`。"赤字≤25%"与"绝对下限"同时生效，取更保守者。触及则封顶并提示，且联动 §8 的 ED 升级处置。

### 2.3 宏量分配 + 冲突消解状态机（P0）
锚定顺序：脂肪（保必需）→ 蛋白（按目标）→ 碳水（补余量）。

| 目标 | 蛋白质 | 脂肪 | 碳水 |
|---|---|---|---|
| 减脂 | 2.0 g/kg | 0.9 g/kg | 余下热量 ÷ 4 |
| 保持 | 1.6 g/kg | 0.9 g/kg | 余下热量 ÷ 4 |
| 增肌 | 1.8 g/kg | 1.0 g/kg | 余下热量 ÷ 4 |

**当热量被安全下限压低、蛋白+脂肪挤占碳水时，按以下确定性流程消解（不得静默输出畸形配比）：**
1. 脂肪先取目标 g/kg，**硬下限 0.6 g/kg**（必需脂肪酸），不再下调。
2. 若 `蛋白×4 + 脂肪×9 > 热量 − 碳水最小值`，则**线性下调蛋白，硬下限 1.4 g/kg**（ISSN 下限）。
3. 碳水最小值 = **50 g/天**（防酮症性不适与脑供能）。
4. 若脂肪=0.6、蛋白=1.4 后碳水仍 < 50 g → 判定该（体重,目标,热量）组合**不可行** → 转"不可行处置"：明确告知"当前热量无法同时满足安全脂肪与足量蛋白"，给选项（降低减脂激进度 / 延长周期 / 提高热量 / 咨询专业人士），**不输出配比**。
- 高 BMI（>30）蛋白/脂肪按**调整体重** `adj = 理想体重 + 0.25×(实际−理想)`，理想体重取 BMI 22 反推（`22×身高m²`）。该调整仅用于宏量 g/kg，不用于 BMR。

### 2.4 膳食纤维（美标，已修订消除低热量矛盾）
- `纤维目标 = clamp(14g/1000kcal × 目标热量, 下限 25g, 上限 50g)`。
- 说明：14g/1000kcal 即 IOM 25g(女)/38g(男) 的同一基准在参考热量下的取值；改用**按实际热量缩放 + 统一 25g 下限**，避免低热量摄入时被 38g 强行抬高的矛盾。50g 为**产品侧舒适上限（防 GI 不适），非 IOM 法定 UL**。
- 纤维不参与热量拆分，推荐引擎取单侧约束（≥ 目标且 ≤ 上限）。

### 2.5 按餐拆分 + 弹性
- 按餐数等比或自定义权重拆分每餐目标。
- **弹性框架（§8）**：目标以"区间"呈现；达成以**周平均**为口径，单日/单餐波动不计为失败。

### 2.6 TDEE 自适应校准（闭环）
- 每 2–3 周用体重移动平均的实际变化反推真实 TDEE，修正 TDEE 乘数（掉秤慢于预期 → 下调）。
- **多目标切换（P1）**：切目标时校准乘数**保留**（它修正个体代谢，与目标无关）；周平均统计窗口重置并提示"切换后约 N 周重新校准生效"。
- 同一校准乘数亦联动调整 §3.1 的运动校正系数 k。

## 3. 运动与同步层

### 3.1 HealthKit 能量口径（P0 已修正双算）
- **实际口径（数据充足时，纯加法）**：`当日TDEE = BMR + k × 实测全天ActiveEnergy`（可选 +TEF）。
  - k = 活动能量高估校正系数，默认 **0.7**（步行/跑步偏准、力量/HIIT 高估更多；按类型分层为 P1，MVP 用统一 0.7，并随 §2.6 校准联动）。
  - 该口径**不使用任何活动系数乘子**，因此从根上不存在"系数 + 实测"的双重计算。
- **计划基线 / 回退**：`BMR×1.2 ± 目标调整`，仅用于晨间排布与**无 Watch 数据**时估计。一旦当日活动数据可用，即用上式"当日TDEE"替换基线、重算剩余餐次额度——**两者择一，从不相加**。
- 注：本版废弃 v2 的 `−BMR×0.2` 与 `×0.5` 串联（两个保守因子叠加导致系统性少回补）。

### 3.2 规划 vs 实测解耦 + 业务日边界（P1）
- 实测活动量累积后，只把增补额度追加到**当天剩余餐次**，不回溯已吃餐次。
- **业务日边界**：以用户本地时区物理日为准，提供可配置"日界点"（默认凌晨 04:00，兼顾熬夜）；HealthKit 数据按设备时区聚合到对应业务日；跨时区旅行以当前本地时区归集。

### 3.3 Rede 集成（解耦）
- Rede 为**可选**数据源（目标、运动消耗）。
- 优先级：用户手动覆盖 > Rede 同步 > App 默认；冲突显式提示，不静默覆盖。
- App 可独立运行，无 Rede 完整可用。

## 4. 食材与数据层（全美标）

### 4.1 原料库（生重）+ 缺失兜底（P1）
- 主源：**USDA FoodData Central**（含 household serving/portion 元数据）。
- 每条目强制字段：每 100g 宏量+纤维、**raw/cooked 标志 + 换算系数**、食材分类、标准份量。
- **USDA 未命中（尤其本地化生鲜）兜底链**：选近似食材 → 用户手动录"每 100g 宏量" → 标记"无数据，不参与优化"。§6.1 明确无数据食材不进入 xᵢ。

### 4.2 包装食品（条形码）+ OCR 五步（P1）
- 链路：扫码 → **Open Food Facts** → 未命中 → **OCR 识别 FDA Nutrition Facts 面板** → 仍失败 → 手动输入；结果写回本地缓存。
- 组件：扫码 VisionKit `DataScannerViewController`（iOS16+）；OCR `VNRecognizeTextRequest`。
- **OCR 五步**（承认是高失败率模块）：① 文本识别 → ② 字段定位 → ③ **份量/单位/分数解析**（美标多为 per serving + "2/3 cup (55g)"，非 per 100g）→ ④ 反推每 100g → ⑤ **强制人工确认**（与"不靠猜"一致）。失败回退手动输入。
- FatSecret 商业条码库列后期增强，不进 MVP。

### 4.3 生熟重换算
- 默认录生重；计算按食材换算系数（米饭 ×~3、瘦肉熟重 ≈ 生重×0.75）。录入确认 raw/cooked。

## 5. 库存模型（v2 核心转向）

> 解决最致命缺陷：精确克级库存摩擦大、录入即漂移，使引擎在虚构数据上运算。

### 5.1 两类食材
- **常备类**：米、面、油、盐、糖、调料 → 默认"无限可用、不追踪、不扣减"。
- **生鲜/限量类**：肉、蛋、奶、蔬果、已开封包装 → 追踪 + 保质期。

### 5.2 轻量库存 + 档位↔克数映射（P1，押注命门）
- 生鲜按"存在性 + 粗档位（充足/少量/快没了）"维护；精确克数仅在用户愿意时填。
- **档位↔克数映射**（供 §6.1 `xᵢ≤库存` 约束使用）：

| 档位 | 约束上界（克数）取值 |
|---|---|
| 充足 | 不设上界（或类的合理上限） |
| 少量 | 取该食材标准份量 × 1–2（保守取下界） |
| 快没了 | ≤ 1 份 |

- 录入：条码 / 拍照 / 预设 / 收据。
- **采纳即扣减**：采纳推荐 → 按用掉克数下调档位（如"充足→少量"）。
- **低频补货确认**：每隔几天 30 秒复核，纠正漂移并重置档位。
- 保质期临近提醒；推荐时优先消耗将过期食材。

## 6. 推荐引擎

### 6.1 优化模型（P0 可行性已具体化）
- 决策变量：各可用食材克数 `xᵢ ≥ 0`（无数据食材不纳入）。
- **MVP 求解：纯线性规划（LP）**——目标为最小化各营养项**加权绝对偏差**（用辅助变量线性化，权重 蛋白 > 热量 > 碳水 > 脂肪 > 纤维）；"1 份水果/份量档位"等整数语义放**后处理**，不进硬约束，避免 MIQP。
- 硬约束：热量 ±7%；蛋白 ≥ 目标 95%；`xᵢ ≤ 库存档位上界`；§6.2 的**单条宽松护栏**（仅防荒谬解）。
- 软约束（容差）：碳水 ±15%、脂肪 ±15%、纤维 ≥ 目标 80% 且 ≤ 上限；**份量合理性软惩罚（见 §6.2）一并计入目标函数**。
- **求解器选型（实现 brief 前置评估）**：纯 Swift 简化单纯形 vs 经许可的 GLPK/Clp 封装——需评估**许可证、二进制体积、on-device 耗时**后定。
- **不靠猜**：LLM 不参与克数计算，仅对已选组合做自然语言解释。

### 6.2 膳食合理性约束（软约束化，v2.2 重审）

> 原 v2.1 把份量做成**绝对硬上限**，问题：① 与硬性宏量约束叠加易致**无解 / 脆弱**；② 固定克数**不随体型 / 目标 / 餐量缩放**，情境失配；③ "份量合理"本质是软偏好，不该当作不可违反的硬规则；④ 越界处理生硬、损害用户自主。改为**软惩罚 + 单条宽松护栏**。

- **软惩罚（进目标函数，主力）**：对"单一食材超出舒适份量"按超出量递增惩罚（建议二次型）——小幅超出代价低、大幅超出代价高，自然劝退"5 斤米饭"而不禁止合理弹性。
- **相对而非绝对**：舒适份量用"占该餐热量的比例"或"标准份量的倍数"表达，随餐量 / 目标 / 体型自动缩放，**不写死克数**。
- **唯一硬护栏（宽松）**：仅保留一条**远高于正常份量**的绝对上限（如任一食材 ≤ 该餐热量 ~70%，或 ≤ 标准份量的某大倍数），纯粹防求解器给出荒谬 / 退化解，**不**用来定义"正常"，故几乎不压缩可行域。
- **餐型模板**：仍鼓励"蛋白 + 主食 + 蔬菜(±水果)"的可吃组合，但作为**软偏好**（缺某类加惩罚），非硬性必须。
- **用户主动加水果**：设为必选 `xᵢ ≥ 1 份`；超出正常份量只受软惩罚牵引回正常量，**不硬性封顶**；用户手动设更大量则允许并给中性提示。
- **可调强度**：用户在"严格 / 宽松"间调节软惩罚权重，无需改代码即可放宽合理性。

### 6.3 无解 → 优雅降级（确定性 fallback）
LP 无解 → 逐步放松软约束 → 仍无解 → 贪心部分解 → 输出"尽力而为的部分解 + **缺口购物清单**"（"还差 30g 蛋白 → 鸡胸/豆腐/希腊酸奶"）。流程图与验收用例见 §15。

## 7. 现实场景兜底

- **外食/无法称重**：off-plan 快速记录（搜索/拍照/估份），系统**重排当天剩余餐次**补偿。
- **临时跳餐/加餐**：实时重算剩余额度。
- **多人共餐**：按"我吃的份额"估记。

## 8. 安全与健康守则（硬要求，含准入门控）

### 8.1 准入门控（P0，onboarding 即拦截）
- **未成年（<18）**：退出标准计算流程，**禁用减脂与任何热量赤字**，仅提供一般均衡膳食信息，强制引导监护人 + 专业人士。
- **孕期/哺乳**：onboarding 设问；命中则**禁用减脂路径**，按增量需求提示并退出标准减脂流程，引导专业人士。
- **异常低体重**：自填数据算出 **BMI < 16** → 二次确认 + **锁定减脂目标**。

### 8.2 热量与宏量安全
- 摄入硬下限 = `max(女1200/男1500, BMR×0.9)`；赤字 ≤25%；二者取严。
- 蛋白上限 ≤3.0 g/kg；脂肪下限 0.6 g/kg；碳水最小 50g（见 §2.3 状态机）。

### 8.3 失调饮食（ED）防护（P0，量化阈值 + 升级处置）
- **触发阈值**：周降幅 >1% 体重 **或** 连续 2 周触下限封顶 **或** BMI < 17.5。
- **升级处置**（非仅提示）：禁用减脂、仅允许保持/增量；显式"暂停追踪 + 专业支持"卡片。
- **资源路由**：按用户地区路由到合适的本地化专业支持入口（不写死单一资源、不对保密性/流程做绝对承诺）。
- 自填体重/目标异常值做合理性校验（BMI<16 触发 §8.1）。

### 8.4 免责
非医疗建议；敏感人群引导专业人士。

## 9. 关键指标与埋点

| 层级 | 指标 |
|---|---|
| 激活 | 档案 → 首份计划转化率；首日库存录入 ≥N 项占比 |
| 留存 | D7 / D30；日均记录餐次；连续记录天数 |
| 核心价值 | 推荐采纳率；营养命中率（落容差内占比）；无解触发率 |
| 库存健康 | 库存新鲜度；"库存外食材"占比（漂移度量） |
| 闭环 | 启用校准用户达成率 vs 未启用 |
| **安全（新增）** | 触下限封顶频次；ED 防护触发率；过快掉秤拦截率；未成年/孕期门控命中率 |

## 10. MVP 范围

**P0**：单位制；准入门控；档案→目标→BMR/TDEE→宏量(含冲突消解)/纤维；安全下限+ED 防护；轻量库存(档位+映射+采纳扣减)；推荐引擎(LP + 合理性约束 + 确定性降级)；HealthKit 加法口径(k=0.7+回退);USDA 原料库 + 条码(OFF) + OCR(含人工确认)；off-plan 快速记录。

**P1**：TDEE 自适应校准；运动校正系数按活动类型分层；餐型模板精修；OCR 容错增强；Rede 深度同步；多目标切换迁移。

**P2**：FatSecret 条码增强；多人共餐；社交/激励。

## 11. 开放假设与风险

- **依从性假设（整盘押注）**：用户接受"低频补货确认 + 采纳即扣减"，拒绝"高频精确称重"。
- USDA 与用户实际食材的匹配度（本地化覆盖）。
- 运动校正系数 k=0.7 需真实数据逐人验证。
- OCR 对多样化包装版式的鲁棒性。

## 12. 数据模型与接口契约（核心 stub，细节留实现 brief）

**核心实体**
- `UserProfile { id, sex, birthDate, height_cm, weight_kg, unitSystem, goal(cut|maintain|bulk), mealsPerDay, bodyFatPct?, flags{isMinor, isPregnantOrLactating}, tdeeCalibrationFactor }`

**关键外部接口（形状）**
- USDA FDC：`GET /v1/foods/search?query=&api_key=` → `{foods:[{fdcId,description,foodNutrients,foodPortions}]}`；`GET /v1/food/{fdcId}`。
- OFF：`GET /api/v2/product/{barcode}.json` → `{product:{nutriments:{energy-kcal_100g,proteins_100g,…}, serving_size}}`。
- HealthKit：`HKStatisticsQuery(activeEnergyBurned, sum, 业务日范围, 本地时区)`。

## 13. 外部依赖与离线策略

- **USDA FDC**：需注册 API key（DEMO_KEY 限频极低），有日配额 → MVP 打包**常见食材本地快照(SQLite)**覆盖高频，API 仅补充。数据为美国政府**公共领域**，可自由再分发。
- **Open Food Facts**：数据为 **ODbL**（含署名 + share-alike 义务）→ **法务需评估随附数据集合规**；API 可用性波动 → 缓存 + 礼貌限频。
- **离线优先**：核心计算与常见食材查询离线可用；联网仅用于条码新品与可选同步。

## 14. 隐私与合规

- HealthKit 用途字符串明确声明、仅用于功能、**严禁用于广告**；隐私政策必备。
- 健康/体重/饮食为敏感数据：本地优先、最小上云、边界清晰。
- App Store 健康类审核清单：医疗免责、数据用途、未成年保护。

## 15. 验收算例（含边界）

| 例 | 输入 | 关键输出 | 验证点 |
|---|---|---|---|
| 正常 | 男30/180/80kg/保持 | BMR=1755；基线2106；P128g/F72g/C236g；纤维≈30g | 宏量回算热量自洽 |
| 运动加法 | 同上，实测 Active=600kcal，k=0.7 | 当日TDEE=1755+420=2175（非基线2106+全额→无双算） | 加法口径、无双重计算 |
| 冲突消解 | 女30/165/90kg/减脂 | 目标≈1555；P/F 挤占致碳水<50g → 蛋白下调至≈1.73g/kg 使碳水=50g | §2.3 状态机生效、未越下限 |
| 不可行 | 极端高体重+激进减脂 | 脂肪0.6、蛋白1.4 后碳水仍<50g → 输出"不可行+选项"，不出配比 | 拒绝畸形配比 |
| 库存无解 | 仅米+蛋+油 | 部分解 + 缺口购物清单（缺蛋白→建议项） | §6.3 降级链 |
