# 原生能力解禁路线图 —— HealthKit / 通知 / 小组件 V1

> **状态:提案 —— 契约进化路线图。每项能力动手前都需架构 owner 逐项批准。**
> 下列三项在 `docs/IRONPATH_MASTER_TECHNICAL_ARCHITECTURE.md` 里目前都是
> **被闸控/被禁止**的(§17 延后、§18 禁止、§6.2 inert stub)。本路线图把它们
> **安全地**排序解禁。每项能力作为一个独立切片激活,其 PR **在同一次改动里
> 修订 §17/§18(及相关条款)**(master §1.1)。在那之前,禁令继续有效。

## 1. 为什么需要这份路线图

你做原生的真实理由里包含 HealthKit、通知、桌面小组件。但当前的绑定契约是
**"纯本地、无 HealthKit、无网络/通知、stub 包保持空壳"**——它当初是为"防 AI
乱改"而防御性写就的,**现在反而挡住了你做原生本该换来的那些能力。** 这份路线图
就是让**契约进化去对齐产品目标**,同时不丢掉数据安全纪律。

## 2. 当前契约里各自被什么挡着

- **HealthKit** —— §6.2:`IronPathHealthKit` 是个空壳 stub、注释明说"不 import
  HealthKit";§17:闸控;§18:禁止填充 stub、禁止引入 HealthKit。
- **通知** —— 未建模;§16/§18 禁网络;§18 禁新增平台依赖。本地通知需要权限项
  (entitlement)+ 授权 UX。
- **小组件** —— 未建模;小组件需要一个**新的 App 扩展 target**(会动
  `project.pbxproj`,平时刻意避开)+ 一个 **App Group** 来共享数据。

## 3. 指导原则(安全主轴不变)

1. **默认仍是 local-first。** 解禁这三项**不**等于打开云、账号或远程推送。
   (远程推送需要服务器 → 继续闸控。)
2. **AppData 仍是源-真值。** 外部数据(HealthKit)**只能经 DataHealth clean
   view 流入 AppData**(§10)——绝不走原始数据、绝不成为第二个源-真值。
3. **每项能力各自带:授权 UX、隐私清单、guard、测试**,并在同一 PR 内修订契约
   (§1.1)。
4. **只走慢车道。** 这三项碰隐私/权限/源-真值,**都不符合**只读 UI 的"快车道"。

## 4. 逐项范围

### 4.1 HealthKit（建议第一个做）

**解锁什么:** 导入体重 + 训练历史;以后可选地把 IronPath 的训练写回 Apple
健康。

**架构:** 把 `IronPathHealthKit` 空壳填成一个真实的"读"适配器。导入的样本
**经 DataHealth clean view** 映射进 `IronPathDomain.AppData`(闸控导入;写前备份;
不伪装成功)。授权由用户控制;**先只读**,写回是以后单独的切片。

**约束:** HealthKit 权限项 + 隐私用途说明(Info.plist / entitlements,很可能动
`project.pbxproj`)、授权 UX、**数据不出设备**。体重存 **kg**(与 `UnitSettings`
一致)。

**修订:** §6.2(stub→激活,带新边界)、§17(HealthKit 闸解除)、§18(HealthKit
从"禁止"变为一个经批准、有边界的适配器)、§8/§10(进 AppData 的闸控导入路径)。

**切片:** `HK-1` 授权 + 读体重 → AppData(闸控);`HK-2` 读训练历史;
`HK-3`(可选、更晚)把 IronPath 会话写回健康。

### 4.2 本地通知（便宜、仍 local-first）

**解锁什么:** 本地提醒——组间休息提醒、训练提醒。

**架构:** `UNUserNotificationCenter`,授权门控,**纯本地调度**。无网络、无远程
推送。

**约束:** 通知授权 UX + 权限项。**远程/推送通知明确排除(需服务器 → 继续闸控
§17)。**

**修订:** §16/§17/§18,只允许**本地**通知;远程推送仍闸控。

**切片:** `N-1` 授权 + 组间休息本地通知;`N-2` 训练提醒(本地排程/取消)。

### 4.3 桌面小组件（需要扩展 target）

**解锁什么:** 一个显示"今日准备度 / 下一次训练"的桌面小组件。

**架构:** 一个 **Widget 扩展 target**(WidgetKit),从共享的 **App Group** 容器
读一份小的**派生**快照(一个本地文件)。小组件**只读**——**绝不**写 canonical
AppData、绝不成为源-真值(它渲染派生视图,和 LocalSnapshot 历史一个道理)。

**约束:** 新的 App 扩展 target + **App Group** 权限项(会动 `project.pbxproj`,
有正当理由)。共享文件是 App 写、小组件读的**派生**快照——仍 local-first、无网络。

**修订:** §5/§12(App Group 作为给小组件用的、经认可的**本地只读**共享)、
§17/§18,并记录新增的扩展 target。

**切片:** `W-1` 准备度小组件(只读、App Group 派生快照)。

## 5. 建议顺序

1. **HealthKit（HK-1 读体重）** —— 产品价值最高;与"数据不丢"的目标协同;只读
   风险最低。
2. **本地通知（N-1 组间休息）** —— 最便宜、仍 local-first、不碰源-真值。
3. **小组件（W-1 准备度）** —— 最后;需要扩展 target + App Group,且要消费前面
   几个面/持久化产出的数据。

> 这三项都排在**核心只读面(快车道)+ 成绩持久化(iOS-17A)落地之后**——小组件/
> HealthKit 在设备上有了真实训练数据后才最有用。

## 6. 待架构 owner 拍板的开放问题

1. **HealthKit 范围** —— 先只读(HK-1/HK-2)、写回(HK-3)延后?还是现在就含写回?
2. **通知** —— 确认**只做本地**(暂不远程推送)?
3. **小组件数据共享** —— App Group + 派生只读快照可接受吗(保持 §12 源-真值不破)?
4. **顺序** —— 接受 HealthKit → 通知 → 小组件,排在快车道 UI + iOS-17A 之后?

## 7. 什么不变

云同步、账号/认证、Supabase/后端、远程推送——**继续闸控**(§17)。原生数据不出
设备(§16)。AppData 仍是唯一源-真值,由 open-bag 保真 + schema/timestamp guard +
DataHealth 保护。解禁 HealthKit/通知/小组件**不是**打开上述任何一项的许可证。

有 guard 地进化去对齐产品。*
