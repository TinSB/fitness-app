# iOS AN-2 — Plateau-Detection Engine Port + Function-Level Parity V1

**Track:** Analytics / Insights (AN-1…6). **Status:** PURE read-only engine port, NOT wired
into any UI (that is AN-6). **Source of truth impact:** none. **Data-safety impact:** none.

## What this slice does

Faithfully ports the per-exercise plateau-detection function `detectExercisePlateau` from
`src/engines/plateauDetectionEngine.ts` (and every private helper it reads) into a new PURE
`IronPathTrainingDecision.PlateauDetectionEngine`, and FUNCTION-LEVEL parity-pins it: the
goldens are GENERATED from the REAL TS engine; the Swift port re-runs `detectExercisePlateau`
on the SAME echoed inputs and COMPUTE-ASSERTS the full `PlateauDetectionResult` == golden.

### Ported surface (line-by-line mirror, TS source lines noted in the Swift comments)

- `detectExercisePlateau` (plateauDetectionEngine.ts:299) — the only public entrypoint.
- All private helpers: `signal` / `unique` / `exerciseIds` / `exerciseMatches` /
  `sessionDateKey` / `relevantSessions` / `relevantExerciseSets` / `firstExerciseName` /

  `summaryByStatus`.
- Output types `PlateauStatus` / `PlateauSignal` / `PlateauDetectionResult` and the input
  `DetectExercisePlateauParams` / internal `SessionPerformance`.

### Dependency boundary (reuse, don't re-port; TYPE-only for trainingLevel / effectiveSet)

- **REUSES** the already-ported engineUtils helpers `E1RMEngine.number` / `setWeightKg` /
  `completedSets` (iOS-17e-1) and the analytics filter `E1RMEngine.filterAnalyticsHistory`
  (the engine's only real cross-module function call), plus the
  `LoadFeedbackEngine.LoadFeedbackValue` canonical value constants (iOS-17e-4). No re-port.
- `e1rmProfile` and `loadFeedback` are genuine TS runtime UNIONS consumed by duck-typing
  (`'recentValues' in profile`, `'counts' in value && 'adjustment' in value`,
  `Array.isArray`, `typeof === 'string'`). The faithful port keeps them as raw `JSONValue?`
  and reproduces the SAME structural discrimination — it never forces a static type the TS
  does not assert.
- `TechniqueQualitySummary` (trainingLevelEngine.ts:41) / `EffectiveVolumeSummary`
  (training-model.ts:1258, consumed as `Partial<…>`) / `PainPattern` (training-model.ts:1028)
  are ported as the consumed TYPE SUBSET ONLY (mirroring the
  `SmartReplacementEngine.SmartReplacementPainPattern` precedent). **No logic** from
  `trainingLevelEngine` / `effectiveSetEngine` is ported — those are AN-3 / AN-5.

### Purity

PURE: consumes `[TrainingSession]` history (a §11 clean input) + an exerciseId + optional
external summaries. No IO, no randomness, **zero `: Date`** — the engine never reads the
wall clock; every date comparison is over the session's OWN date strings (`localeCompare`
descending, reproduced with `>` over ASCII ISO keys, the same precedent as
`E1RMEngine.collectCandidates`). Not wired into any decision output or UI.
