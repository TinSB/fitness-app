# SR-2 — Replacement Engine Port V1

Faithful Swift port of the **underlying replacement engine** in
`src/engines/replacementEngine.ts` into the pure `IronPathTrainingDecision`
package, parity-pinned against script-generated goldens. Part of the SR engine
porting track: **SR-1** ported the exercise library (data + parse functions);
**SR-2** (this milestone) ports the replacement logic that consumes it; the
top-level `smartReplacement` engine remains **SR-3 (deferred)**.

Binding rules live in `docs/IRONPATH_MASTER_TECHNICAL_ARCHITECTURE.md`
(§19.2 engine-package pure logic + parity goldens; §27 milestone row). This
file is descriptive context.

## Goal

Port the three replacement-engine functions and the "engine knowledge" they
read into Swift, **byte-faithfully**, and lock the port to TypeScript behaviour
with generated parity goldens that the Swift port reproduces item-by-item.

Into **`ReplacementEngineKnowledge.swift`** (pure data; the "engine knowledge"

### Faithfulness notes

- `normalizeName` (replacementEngine.ts:166) uses `/[（）()\\s-]/g` — a **double
  backslash** in a regex literal, i.e. the char class `{ （ ） ( ) \ s - }`. It
  strips a literal backslash and the literal lowercase `s` (NOT whitespace) and
  lowercases AFTER. Reproduced exactly (a Swift character-set filter then
  `lowercased()`).
- JS `Array.prototype.sort` is stable; Swift `sorted(by:)` is not — so the rank
  sort breaks ties on the pre-sort index and the equipment re-sort breaks final
  ties on the captured `equipmentSortIndex`, matching the stable JS sorts.

## Non-goals (hard boundaries)

- **NOT** the session-mutating `applyExerciseReplacement` /
  `restoreOriginalExercise` — those are the **WRITE PATH** and stay unported.
- **NOT** the top-level `smartReplacement` engine (`smartReplacementEngine.ts`)
  — that is **SR-3**. The 4 SR-0 `smart-replacement/*` goldens are untouched.
- **NO** change to existing engine logic, existing goldens, source-of-truth, the
  write path, `project.pbxproj`, `package.json`, or the lockfile. Zero `: Date`,
  no IO, no clock, no randomness.

## Parity mechanism

`scripts/parityGoldensEntry.ts` gains two generators driven by the REAL engine:

- `generateReplacementEngineKnowledge` → `replacement-engine/knowledge-snapshot-v1`:
  dumps the chains + the engine-used override subset; `ReplacementEngineParityTests`
  reconciles the ported Swift tables against it **item-by-item** (key sets, every
  field, array order, counts 60 / 63).
- `generateReplacementEngine` → 4 OUTPUT goldens: runs `buildReplacementOptions`
  over a fixture's `exercise` + `context`, echoes the `engineInput`, and pins the
  full `ReplacementOption[]` + `validation` / `synthetic` probe verdicts. The
  Swift test re-runs the PORTED engine on the SAME echoed input and asserts
  equality — true line-by-line parity, not a shape check.

Goldens are GENERATED by `scripts/generate-parity-goldens.mjs`, **never
hand-edited** (§22). Running it reports `generated 24 fixture(s); 5 changed`; the
19 pre-existing goldens regenerate byte-identically.
