# iOS-17e — Engine Consumption of Performed Sets (Closed Loop) — Plan V1

> **Status:** 17e-0 (parity scaffold) + 17e-1 (per-exercise e1RM) + 17e-2 (performance query) shipped; 17e-3…17e-5 proposed (approved track).
> **Scope of 17e-0:** parity *baseline* only — **no engine logic ported.**
> It adds history-bearing decision fixtures, generates their adaptive goldens,
> and DECODE-ONLY-pins them on the Swift side. The progression-cluster port
> lands incrementally in 17e-1…17e-5.

iOS-17 shipped native per-set logging: performed sets (weight/reps/RIR) are
written to canonical `AppData.history[].exercises[].sets[]` through the one
sanctioned `performGatedMutation` write path (iOS-17A #426; corrections via
DEEP-EDIT-1). The **closed loop is not yet closed**: the native
`TrainingDecision` engine does not *read that performed-set history back* to
adapt the prescription. iOS-17e closes it — the engine consumes logged results
and emits a history-driven, per-exercise progression — ported as a sequence of
parity-pinned slices, the same discipline the original TrainingDecision port
used (iOS-4B0…4B5).

## 1. The gap — static native template vs PWA adaptive progression cluster

**Native (today).** The Swift `IronPathTrainingDecision` engine produces a
**static** exercise prescription. `TrainingDecisionExercisePrescription` is
explicitly **DEFERRED-inert** (it carries no performance-derived progression),
and the strength signal is a single **global boolean** e1RM trend
(`TrainingDecisionE1RMTrend`, a global top-set pool — *not* per-exercise; see
the §11 SR-4 note). The performed sets the user logs do not change the next
prescription's load.

**PWA (source of truth).** The TypeScript engine has a **performance-adaptive
progression cluster** (~1420 lines) wired deep into
`src/engines/trainingDecisionEngine.ts`:

| TS module | Role |
| --- | --- |
| `progressionRulesEngine.ts` | per-exercise progression decisions (add load / hold / back off) |
| `e1rmEngine.ts` | per-exercise estimated-1RM trend (not a global boolean) |
| `adaptiveFeedbackEngine.ts` | adaptive deload / recovery decisions from logged feedback |
| `setWeightFineTuneEngine.ts` | per-set target-weight fine-tuning |
| `loadFeedbackEngine.ts` | load-feedback risk signal feeding the weekly adjustment |

These drive the engine's **`perExercise`**, **`weeklyAdjustment`**, and
**`progressionMode`** outputs. With a history that shows progression
(`isE1rmTrendUp` over ≥ 4 completed sessions), the real TS engine returns
`progressionMode='progress'` + `weeklyAdjustment.direction='increase'`; with a
flat or too-short history it holds. The native engine cannot reproduce this yet.

iOS-17e ports this cluster across **17e-0…17e-5**.

## 2. Slice decomposition (17e-0 → 17e-5)

| Slice | Scope | Parity posture |
| --- | --- | --- |
| **17e-0** | **Progression parity scaffold (this slice).** Add history-bearing decision fixtures + generate adaptive goldens + DECODE-ONLY Swift pins. **No engine logic.** | Cold-start goldens **zero drift**; new history goldens **decode-only**. |
| **17e-1** | Per-exercise e1RM — port `e1rmEngine` (replace the global boolean with a per-exercise estimated-1RM trend). | Flip the e1RM-dependent fixture(s) decode-only → compute-assert. |
| **17e-2** | Performance query — the per-exercise history lookups the progression rules read (top-set pool, recent-session selection). | Compute-assert the per-exercise performance projection. |
| **17e-3** | Progression suggestion — port `progressionRulesEngine` (add-load / hold / back-off per exercise → `progressionMode`). | Flip `progressive-overload` / `plateau-stall` / `insufficient-history` to compute-assert. |
| **17e-4** | Fine-tune / load feedback — port `setWeightFineTuneEngine` + `loadFeedbackEngine` (per-set target-weight + load-risk → `weeklyAdjustment`). | Compute-assert the fine-tuned targets + weekly adjustment. |
| **17e-5** | Wire to the top surface — surface the adaptive per-exercise prescription in the app (read-only display; the gated write path is unchanged). | End-to-end compute-assert; the scaffold is fully consumed. |

Each slice is an **engine-package pure-logic + parity** change (Allowed-change
§19.2). The write path (`CanonicalSessionWriter` / `performGatedMutation`) is
**never** touched — 17e only makes the engine *read* logged history, never
write.

## 3. Parity strategy — cold-start zero drift, history fixtures, decode-only → compute-assert

The whole track rests on three parity rules:

1. **Cold-start goldens stay byte-identical (zero drift).** The 10 existing
   `training-decision/*` goldens (and the other 14 parity goldens) regenerate
   unchanged at every slice. The `--diff-filter=MD` guards (`changedFiles(...)`)
   ignore pure *additions* but flag any *modify/delete* of an existing golden,
   so a fixture-adding slice stays green while drift of a committed golden fails.
   The new ids are **NOT** added to `TrainingDecisionGoldens.allIds` /
   `.expandedIds`, so every existing compute/decode parity test stays pinned to
   the 10 cold-start goldens.

2. **History fixtures carry performed sets.** The new
   `inputs/training-decision/*` fixtures set `synthetic.sessions[].sets`
   non-empty, so the REAL engine's `isE1rmTrendUp` / progression path runs and
   the generated golden captures the adaptive `progressionMode` /
   `weeklyAdjustment` / `perExercise` output. They are GENERATED by
   `scripts/generate-parity-goldens.mjs`, **never hand-edited** (§22), and
   reuse the existing `generateTrainingDecisionExpanded` projection (no new
   golden shape).

3. **Decode-only now, compute-assert as the engine lands.** Because the static
   native engine cannot reproduce an adaptive golden, 17e-0 asserts only that
   the goldens **decode into the parity `TrainingDecision` shape** and that the
   shape is **stable** (`TrainingDecisionProgressionGoldenDecodeTests`). It does
   **not** compute-assert (which would require the unported progression engine).
   As 17e-1…17e-5 port each layer, the corresponding fixture(s) flip from
   decode-only here to a real Swift-engine **compute-and-assert** — the exact
   SR-0 → SR-3 decode→compute progression.

### 3.1 The 17e-0 fixtures

| Fixture | History | Adaptive golden output | Pins |
| --- | --- | --- | --- |
| `progressive-overload-v1` | 5 completed sessions, rising weight 60→70 kg, neutral readiness | `progressionMode='progress'`, `weeklyAdjustment.direction='increase'` | the **closed loop**: rising performed sets → recommend adding load |
| `plateau-stall-v1` | 5 completed sessions, flat weight 60 kg | `progressionMode='hold'`, `direction='hold'` | no upward trend → hold (the stall signal) |
| `insufficient-history-v1` | 2 completed sessions, rising weight (< 4-session threshold) | `progressionMode='hold'`, `direction='hold'` | data-insufficient boundary → no premature progression |

The three isolate the progression signal: identical cadence, only the weight
trend / sample count differ, and only the rising-and-sufficient one progresses.

## 4. Discipline (§11.3 / §22) + hard red lines

- **§11.3 — output shape is parity-pinned.** 17e changes the engine's adaptive
  output as it ports each layer. Every such change **regenerates the goldens via
  the generator and reviews the diff** — it never edits a golden by hand and
  never changes a golden shape without a generator change behind it.
- **§22 — goldens are generated.** All `training-decision/*` goldens come from
  `node scripts/generate-parity-goldens.mjs`. The fixture-count guards
  (`parityFixturesContract`, `parityFixturesGenerationConsistency`,
  `iosBootstrapParityStillGreen`, the `iosLocal*` / `iosNative*` `--check`
  guards, and the per-slice `readdirSync(... training-decision).length` guards)
  bump in sync with every additive slice (17e-0: 24 → 27 total; 10 → 13
  training-decision).
- **Hard red lines (stop and escalate if hit):**
  - **Pure engine, no write.** 17e only makes the engine *consume* logged
    history. It never touches `CanonicalSessionWriter`, `performGatedMutation`,
    the write path, or the AppData source of truth. No `schemaVersion` bump.
  - **No `: Date`.** The engine stays deterministic with an injected clock
    (`deterministicClockIso`); no ambient clock / randomness (§11.2).
  - **No `project.pbxproj` / `package.json` / lockfile edit.** New Swift tests
    are SPM-auto-included; no new npm dependency.
  - **Cold-start parity unbroken.** The static engine stays all-green on the 10
    cold-start goldens; the new adaptive goldens are decode-only until their
    slice ports the matching engine logic.
