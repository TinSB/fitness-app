## 1. Goal

Expand `tests/fixtures/parity/inputs/training-decision/` and
`tests/fixtures/parity/golden/training-decision/` from a single fixture
(`normal-session-v1`, which only exercises the reentry-productive path) to a
**10-fixture suite** that locks every load-bearing TrainingDecision V2 path
with deterministic, byte-stable golden baselines — so the future Swift port
(iOS-4B+) has enough golden coverage to be safe.

iOS-4B0 adds **9 new synthetic fixtures**, each driving the REAL Clean Input
Contract pipeline (`buildCleanAppDataView` → `createCleanTrainingDecisionInput`
→ `buildTrainingDecisionFromCleanInput`) through a distinct engine path.

## 2. Why iOS-4B0 exists before Swift implementation

iOS-4A's CodeGraph audit found that `training-decision/normal-session-v1` —
despite its name — actually exercises the **reentry-productive** path (gap-based
phase + low-data progress) and serialises only `decisionVersion`, `userFacing`,
and `hiddenDebugSignals.arbitrationTrace`. Twelve distinct engine branches were
documented as uncovered. Porting `buildTrainingDecision` to Swift against a
single golden would leave severe-rest, controlled-reload, deload, restart, the
stale-data gates, the no-all-1-set regression lock, and the legacy-advice /
clean-input contracts entirely unverified — a high-risk port.

iOS-4B0 front-loads the golden baselines so the Swift port is a
match-the-golden exercise, not a guess.

## 8. Input envelope shape

Every new input fixture is a compact declarative spec (NOT a serialised
AppData), keeping the committed file small and synthetic:

```jsonc
{
  "parityMeta": {
    "id": "training-decision/<name>-v1",
    "schemaVersion": 8,                       // === STORAGE_VERSION
    "describes": "...",
    "privacy": "synthetic",
    "generatedFrom": "scripts/generate-parity-goldens.mjs",
    "tsCommit": "97074a0a662b53c49011ce7e65e84f0670e3bfc7",
    "generatedAtPolicy": "deterministic-clock",
    "deterministicClockIso": "2026-05-27T10:00:00.000Z",
    "sourceNotes": "..."
  },
  "synthetic": {
    "selectedTemplateId": "push-a",
    "trainingMode": "hybrid",
    "sessions": [ { "id", "daysAgo", "sets":[{weight,reps}], "legacyAdvice?", "dirty?": "lifecycle|duration" } ],
    "todayStatus": { "daysAgo", "sleep?", "energy?", "soreness?" },
    "healthMetricSamples?": [ { "daysAgo", "type", "value", "unit" } ],
    "settings?": { "healthIntegrationSettings": { "useHealthDataForReadiness": true } }
  },
  "decisionMetadata": {
    "templateId": "push-a",
    "trainingMode": "hybrid",
    "acutePainReported?": true,
    "explicitDeloadAssigned?": true,
    "useHealthDataForReadiness?": false
  }
}
```

The generator's `buildSyntheticAppData(spec, nowIso)` materialises this into an
engine-valid `AppData` via `tests/fixtures.ts` `makeAppData` / `makeSession` /
`makeStatus` (all dates computed relative to the deterministic clock).

## 9. Golden output shape (enriched)

`generateTrainingDecisionExpanded()` emits structured engine fields (not just UI
text) so the Swift port has machine-checkable baselines:

- `decisionVersion`, `sessionIntent`, `activePhase`, `volumeMode`,
  `intensityMode`, `progressionMode`, `riskLevel`, `trainingMode`
- `weeklyAdjustment`, `finalVolumeMultiplier`, `exerciseRoleFloors`,
  `weeklyBlockReasons`, `progressClarityTripletSuppressed`
- `effectivePhase: { activePhase, gapDays, mode, severity, overridden, hasHistory }`
- `perExercise: [{ exerciseId, role, targetSets }]`, `allTargetSets`,
  `minTargetSets` — set-count regression detectors
- `userFacing` — all 7 surfaces (full)
- `hiddenDebugSignals.arbitrationTrace`
- `cleanInput: { cleanViewBuilt, useHealthDataForReadiness, diagnostics{ lifecycleResidueSessionIds, legacyAdviceSessionIds, invalidDurationSessionIds, cappedIssueScoreKeys, staleTodayStatus, staleHealthData, filteredPerformanceDropIds } }`
- `inputEvidence: { historyLength, healthMetricSampleCount, rawHealthSamplesPreserved }`

The aux fields `finalDose / decisionCategory / decisionStrength /
arbitrationReasonCode` (referenced by the legacy `generateTrainingDecision` and
stripped as `undefined`) are **deliberately not referenced** by the expanded
generator — iOS-4A confirmed they do not exist on `TrainingDecision`.

## 10. Deterministic clock strategy

All 9 fixtures use `deterministicClockIso = "2026-05-27T10:00:00.000Z"` (matching
the existing fixture). History session dates, todayStatus dates, and health
sample timestamps are all computed as `clock − N days` inside the generator, so
gaps (reentry/restart) and staleness windows (todayStatus 3d, health 14d) are
exact and reproducible. No `Date.now()` / `Math.random()` anywhere. Two
consecutive `node scripts/generate-parity-goldens.mjs` runs produce byte-identical
goldens (verified: `--check` reports 0 changed).

## 11. Privacy rules

- `privacy: "synthetic"` for all 9. No real userId / email / token / device label.
- Input fixtures carry ONLY the declarative `synthetic` spec — never a
  serialised AppData. The `DEFAULT_*` constants `makeAppData` injects
  (`userId: 'local-user'`, etc.) live only in the in-memory AppData and are
  never written to a committed file.
- The generator's privacy guard runs on input text + golden text and would abort
  on any `userId/deviceLabel` literal outside the allowlist; both pass.
- A dedicated test asserts no new fixture/golden contains a private identifier.

## 15. Risks

| # | Risk | Mitigation |
| --- | --- | --- |
| R1 | Golden drift if a TS engine changes before the Swift port | `--check` is a CI drift detector; goldens carry `tsCommit`; the Swift runner must match it |
| R2 | `makeAppData` `DEFAULT_*` userId leaking to a committed file | Compact-spec design never serialises AppData; privacy guard + test enforce |
| R3 | Generator coupling to `tests/fixtures.ts` | Acceptable — both are test infrastructure; a change in `makeAppData` would surface as a `--check` diff |
| R4 | deload-week trace is thin (`AR-5` only, since `explicitDeloadAssigned` routes weeklyDirection→decrease) | Locked via `sessionIntent='deload-week'` + `volumeMode='trim'`, which are the distinguishing signals |
| R5 | stale-today-status does not alter the decision (the guard only flags, does not strip todayStatus) | Golden locks the actual TS behaviour: `diagnostics.staleTodayStatus=true` with neutral status values so no false signal arises |

## 16. Final verdict

The TrainingDecision golden suite now covers **10 distinct paths** with
deterministic, byte-stable, privacy-clean baselines, all driven through the real
Clean Input Contract pipeline. The existing `normal-session-v1` golden is
unchanged. `--check` is green, the full TS suite is green, and no Swift /
package / lockfile surface changed. The Swift port (iOS-4B+) now has sufficient
golden coverage to be a match-the-golden exercise. **Recommendation: proceed to
iOS-4B1.**
