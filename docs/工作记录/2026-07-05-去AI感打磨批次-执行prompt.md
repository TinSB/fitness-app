- 行为改动走标准闭环：能进包测试的先写**失败测试**（红）→ 最小实现（绿）→ 回归；纯 SwiftUI 布局不可单测的，以模拟器实拍前后对比为验收证据（app target 无测试 target 是既有边界，如实标注）。
- 每个 PR 合并前必过：`.claude/quality-gate.cmd` 全绿 + 独立 code-reviewer 子 agent 审 diff + 模拟器实拍验收截图（中英双语各一张，涉及双语文案时）。
- 文案改动必须中英成对过文案基线（docs/REDE_PRODUCT_COPY_BASELINE.md）：UI 短语无句号、去 AI 味三铁律（冷/准/短）、两语各自母语表达不逐句机翻；新串同步 RedeL10n golden/禁词测试。
- **§3.4 周口径防回潮测试必须全绿**：排期顺序词「接下来/再往后 · Coming up/After that」是拍板措辞，T2 折叠布局但不许动摇口径。
- **ForgedCard 预算（设计语言 §12.6）**：每屏常驻铭牌至多 1 张、门禁强制。本批一律优先用 surface 原语（`RoundedRectangle` + redeSurface，参照 PlanTabView adjustmentSection 的 0 卡做法），**不动门禁预算数字**；如实现中判断必须用 ForgedCard 且需改预算 → 停下等 owner（门禁改动属红线）。
- 全部任务**不碰** AppData canonical schema、不碰训练历史写路径、不碰引擎裁决逻辑（T6 仅允许 preferences 文件加向后兼容 key）。越界即停下报告。
- auto-merge 仅限门禁全绿 + 审查无 MAJOR + 纯 UI/文案范围（owner 常设授权）；任一存疑只开 PR 等 owner。同一任务修到第 3 次不对 → 停（三振出局）。
- 每个 PR 合并后追加 DEV_LOG 条目（给不读代码的 owner：现象/为什么要紧/改了什么/怎么证明）。
- 批次收口：规格写回（见每任务「规格写回」行 + 文末汇总表）；出批次战报 DEV_LOG 条目。

模拟器验收通用钩子（RootTabView/TodayTabView/PlanTabView）：`-locale zh|en`、`-initialTab train|progress|plan`、`-forceOnboarding`、`-skipOnboarding`、`-autoStartSession`、`-autoCompleteSession`（练完态直达）、`-autoAdvanceExercises N`、`-autoPartialSession`、`-autoResumeAction resume|discard|later`、`-autoOpenSharePreview`、`-autoOpenSettings`。真计划态：`-forceOnboarding` 用 computer-use 点四问（肌肉/4 天/商业健身房/练过）生成，再 `-initialTab plan`。收尾卸载测试 App 不留假数据。

## Tasks（按此顺序执行）

### T3: 今日页首练 "first" 徽标去重
- **现象**：首练时顶部副标题已说「首次训练，全部动作从轻起步」（TodayEngineCopy.swift:156），列表 8 行右侧仍逐行挂「首练/first」徽标（ListCopy.swift:56 `firstTimeShort`），同一信息重复 9 次，数据表 dump 感。
- **scope**：当**全部**动作均为首次时，行级徽标整列隐藏（顶部句总括）；仅**部分**动作首次时（换入新动作场景），保留那几行的徽标（徽标此时有区分价值）。判定逻辑（all-first）抽到可测处或以最小 view 层条件实现。
- **files likely affected**：`ios/Rede/TodayTabView.swift`（行 trailing 渲染 + all-first 条件）；`firstTimeShort` 串本身不动。
- **acceptance criteria**：首练今日页 0 个行级 first 徽标、顶部首练句在；正常（非首练）今日页无变化；换入单个新动作时该行徽标仍显示。
- **tests**：判定逻辑如落在可测层则加单测；否则如实标注 app target 无测试 target，以实拍为证。
- **validation**：门禁 → 模拟器 `-skipOnboarding` 首练今日页中英实拍（改前后对比）。
- **data safety**：零数据接触（纯渲染条件）。
- **merge**：可 auto-merge。**depends on**：无。
- **规格写回**：PRD §5.1 FR-T2 验收标准补一句「首练时首练标记不逐行重复」；不新增 FR。

### T6: 训练卡教学文案一次性化
- **现象**：训练卡永久驻留两行说明——「点重量可调整　之后的建议随之更新」（TrainEngineCopy.swift:73，纯教学）与「按计划目标开始」（:53，anchor 说明）。说明书印在界面上 = demo 感。
- **scope**：① `trainAdjustHint`（:73）改一次性：仅在用户**从未调整过重量**且**累计完成训练 < 3 场**时显示；一旦调过重量或满 3 场，永久不再出现。记忆落 SessionStore preferences 文件（加向后兼容 key，缺省=未见过；**不碰 canonical app-data schema**）。② `trainAnchorNote`（:53）先查消费逻辑：若为多态状态行（会随调整变为「跟随你的调整」等）则**保留不动**（它是状态信息非教学）；若恒定纯教学则并入 ① 同策略。执行时以真代码为准并在 PR 描述写明判定依据。
- **files likely affected**：`ios/Rede/TrainTabView.swift`（显示条件）；`ios/Rede/SessionStore.swift`（preferences key）；串本身不动。
- **acceptance criteria**：新用户前 3 场可见提示；调整过一次重量后立即消失且重启不复现；老用户（≥3 场历史）不见提示；anchor note 行为按 ② 判定结果。
- **tests**：显示判定条件如可抽逻辑层则单测（场数/已调整/组合边界）；preferences 读写向后兼容（旧文件无 key 不崩、缺省语义正确）。
- **validation**：门禁 → 实拍三态：首训显示 / 调重量后消失 / `-autoCompleteSession`×3 后新训练不显示。
- **data safety**：仅 preferences 加 key（向后兼容）；训练历史/canonical schema 零接触。
- **merge**：可 auto-merge。**depends on**：无。
- **规格写回**：Copy Baseline 对应条目标注「一次性提示」策略；System Logic 训练页节补显示条件一句。

- **acceptance criteria**：计划页每个训练日类型的模式构成全文只出现一次；8 场顺序仍一眼可见；点类型行（或等价入口）仍进 FR-PL6 编辑器且编辑后排期如实刷新；顺序编辑入口在；周口径防回潮测试绿；4 天与非 4 天模板中英实拍无破版。

### T1: 练完态当日总结卡 + 分享入口（价值最高，压轴）
